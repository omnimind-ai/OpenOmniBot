# Project Soul Guide

Complete format and maintenance protocol for `PROJECT_SOUL.md` and the field schema / data layout extensions to `PROJECT_CONTEXT.md`.

---

## PROJECT_SOUL.md — Purpose

`PROJECT_SOUL.md` is the behavioral contract for a Workbench Project. It is written by the agent and updated as the user expresses preferences, rules, and constraints. It is **not** auto-generated by the runtime.

**Read it before:**
- Every hot update (so style/behavior changes respect existing rules)
- Every agent task execution that involves business logic (categorization, filtering, notifications)
- Any time the user asks "why did you do X?" — the soul explains the intent

**Write/update it when:**
- Creating a project (extract rules from creation intent)
- User states a new rule: "以后奶茶算娱乐"
- User expresses a preference: "不要暖色系"
- User corrects behavior: "你不应该这样分类"
- After a significant hot update that changes core behavior

---

## PROJECT_SOUL.md — Full Format

```markdown
# <Project Name> — Project Soul

## 创建意图

<One or two paragraphs. What the user said they wanted, their goal, context, and
who they are. Write it as if briefing a new team member on why this project exists.>

Example:
用户想管理个人日常支出，主要记录餐饮、交通和购物。目标是月底能看到
分类汇总，找出花钱最多的类别。不需要收入记录，只追踪支出。

## 业务规则

Rules the user has explicitly stated or that are clearly implied by the domain.
Each rule is one bullet, written as a constraint the agent must enforce.

- <Rule>: <reason or source if known>

Example:
- 奶茶、咖啡归类「娱乐」，不归「餐饮」（用户2026-05-13说的）
- 单笔超过500元时必须填备注，否则拒绝写入并提示用户
- 不记录收入，创建记录时 amount 必须为正数（支出）
- 外卖和堂食都归「餐饮」，不拆分
- 如未说明 category，默认「其他」，不猜测

If no rules yet:
（暂无明确规则，随用户使用中逐步补充）

## 字段约束

Domain field types, required/optional status, valid values, and default logic.
Only list fields that have real constraints. Omit unconstrained free-text fields.

| Field    | Type    | Required | Valid values / constraints       | Default         |
|----------|---------|----------|----------------------------------|-----------------|
| amount   | number  | yes      | 正数，单位人民币元，最多两位小数 | —               |
| date     | date    | yes      | YYYY-MM-DD                       | 今日            |
| category | enum    | no       | 餐饮/交通/购物/娱乐/医疗/其他   | 其他            |
| note     | string  | no       | 商户名或自定义备注，不超过100字  | —               |

Type vocabulary: `string`, `number`, `boolean`, `date`, `datetime`, `enum:<a>|<b>|<c>`, `array`, `object`.

## 显示偏好

User-expressed preferences for the visual presentation. Leave blank if not stated.

- 列表排序: 按 date 倒序，同日按录入时间倒序
- 主色调: 蓝色 (#007AFF)，不使用暖色系
- 金额显示: 保留两位小数，加「元」单位
- 空状态: "还没有记录，点击上方按钮开始记第一笔"
- 分类图表: 饼图（Chart.js），不用柱状图

If no preferences stated:
（无特殊显示偏好，使用 base.css 默认样式）

## 禁止行为

Things the agent must never do in this project, regardless of user requests in the moment.
These represent hard constraints from the domain or from past corrections.

- 不得生成模拟数据、占位数据或示例数据（所有数据来自用户实际操作）
- 不得在用户未确认的情况下删除或归档已有记录
- 不得修改 data/items.json 路径以外的持久化文件
- 不得在 HTML 中硬编码分类列表（必须从 item.fields.category 聚合）

Add domain-specific items as the user expresses them.

## 更新历史

Append one line per change. Oldest first.

- 2026-05-13: 项目创建，初始规则从用户意图提取
- 2026-05-14: 用户说"奶茶算娱乐"，更新业务规则
- 2026-05-15: 热更新：添加月度图表，更新显示偏好
```

## PROJECT_CONTEXT.md — Field Schema Extension

The runtime auto-generates field schema with weak types (all `string`). After creation, overwrite the field table with real types using `file_edit`.

### Type vocabulary

| Type token | Meaning | Example |
|---|---|---|
| `string` | Free text | `note: string` |
| `number` | Numeric, int or float | `amount: number` |
| `boolean` | true / false | `is_recurring: boolean` |
| `date` | YYYY-MM-DD | `date: date` |
| `datetime` | ISO 8601 | `created_at: datetime` |
| `enum:a\|b\|c` | One of fixed values | `category: enum:餐饮\|交通\|购物` |
| `array` | JSON array | `tags: array` |
| `object` | Nested JSON | `meta: object` |
| `?` suffix | Optional | `note: string?` |

### Example field table (expense tracker)

```markdown
## Item Fields Schema

| Field    | Type              | Required | Constraints / Notes              |
|----------|-------------------|----------|----------------------------------|
| title    | string            | yes      | Item display label (商户名简称)   |
| amount   | number            | yes      | 正数，单位人民币元，≤ 2 小数位   |
| date     | date              | yes      | YYYY-MM-DD，未填取今日           |
| category | enum:餐饮\|交通\|购物\|娱乐\|医疗\|其他 | no | 默认「其他」 |
| note     | string?           | no       | 商户名或备注，≤ 100 字           |
| source   | enum:手动\|截图导入 | no     | 录入来源，默认「手动」           |
```

## PROJECT_CONTEXT.md — Data Layout Section

Append after the HTML Element Inventory section. Replace `{spacePath}` with the actual path from `project.spacePath`.

```markdown
## Data Layout

All paths relative to `{spacePath}` (e.g. `/workspace/projects/oob-workbench-expense`):

| Path | Content | Who manages it |
|---|---|---|
| `data/items.json` | All items: active + archived. Never write directly. | Runtime |
| `PROJECT_CONTEXT.md` | This file: API contract, field schema, element IDs. | Agent — update on changes |
| `PROJECT_SOUL.md` | User rules, constraints, display preferences. | Agent — update on user input |
| `backend/api_spec.json` | API specs mirror. | Runtime |
| `backend/scripts/` | Python scripts for `run:{use:"script"}` APIs. | Agent — write via `file_write` |
| `frontend/page_spec.json` | Frontend spec mirror. | Runtime |
| `frontend/html/` | HTML, CSS, JS display files. | Agent — via `workbench_project_update` |
| `frontend/html/base.css` | Standard mobile CSS baseline. | Agent — copy from skill assets on creation |
| `exports/` | Files generated by scripts (CSV, PDF, JSON reports). | Agent scripts |
| `logs/api_calls.jsonl` | Per-call execution log (append-only). | Runtime — read-only |
| `logs/hot_updates.jsonl` | Hot update log. | Runtime — read-only |
| `logs/progress.jsonl` | Creation/update progress events. | Runtime — read-only |

Script I/O convention:
- Input: JSON on stdin → `{"projectId":"...", "inputs":{...}, "items":[...]}`
- Output: JSON on stdout → `{"success":true, "outputs":{...}}` or `{"success":false, "errorCode":"...", "errorMessage":"..."}`
- Write exports to `{spacePath}/exports/<filename>` and return `filePath` in outputs.
```

## Maintenance Protocol

### On creation (Phase 4.5)

1. Write `PROJECT_SOUL.md` — extract intent + rules from user message + domain common sense
2. Read auto-generated `PROJECT_CONTEXT.md`
3. Replace field schema table with real types
4. Append Data Layout section

### On hot update (before writing any HTML or API changes)

1. `file_read({spacePath}/PROJECT_SOUL.md)` — check for rules that constrain the change
2. `file_read({spacePath}/PROJECT_CONTEXT.md)` — check API IDs and element anchors
3. After the update, append one line to PROJECT_SOUL.md 更新历史 if behavior changed

### When user expresses a rule

Pattern: "以后 X 算 Y", "不要 Z", "我希望 A", "你不应该 B", "记住 C"

1. Identify which section of PROJECT_SOUL.md is affected
2. `file_edit({spacePath}/PROJECT_SOUL.md, oldText, newText)` — insert the new rule
3. Confirm to the user: "好的，我已经记住了：[rule]。以后都会这样处理。"

### When APIs or fields change

After any `workbench_project_update` that adds/removes APIs or changes field definitions:

1. Update the API Contract table in PROJECT_CONTEXT.md
2. Update the Item Fields Schema table if fields changed
3. Update the HTML Element Inventory if `data-oob-id` elements were added/removed

## Anti-patterns

**Do not put in PROJECT_SOUL.md:**
- Implementation details (which HTML file to edit, which function to call)
- Temporary state ("user is currently on step 2 of onboarding")
- Git history or change diffs

**Do not put in PROJECT_CONTEXT.md:**
- User rules or preferences — those go in PROJECT_SOUL.md
- Motivation or narrative — PROJECT_CONTEXT.md is a reference table, not prose

**Do not leave PROJECT_SOUL.md empty after creation.** Even if the user said nothing explicit, extract domain-common rules (e.g. for a fitness tracker: "体重必须为正数", "日期不能是未来日期") and write them as initial rules with the note "（从领域常识推断，非用户明确说明）".

Agent Stream + RunLog 端侧优化实现报告

日期：2026-05-17 分支：wzw-dev 目标：流输出降载 + RunLog 可复用 + GUI Agent 外部接入

结论

当前实现把 Agent 流输出、RunLog 收集、可复用 Function、OmniFlow 端侧执行和外部 GUI Agent MCP 触发拆成稳定的五层。 高频 token 和工具预览走合并与帧级批处理；结构化工具生命周期保序；RunLog 使用 append-only 事件流保证审计和恢复； 成功 RunLog 可以编译成本机可重放 Function，纯端侧动作优先走 model-free 执行，无法确定的感知和 data-flow 步骤保留 agent fallback。

端侧流优化LLM content/tool preview 做 conflation，Android 主线程按帧 flush，Flutter 只重绘变化区。
RunLog 机制NDJSON 事件日志 + 节流 snapshot，timeline 可从 snapshot 后的事件重放恢复。
Replay 机制RunLog card 编译为 oob.reusable_function.v1，OmniFlow action alias 统一成 canonical action。
外部接入Python AgentKit 支持 repo probe、prompt bundle、MCP list/guard/run 验收。

到底怎么实现

1. 流输出降载

LLM SSE chunk
Accumulator 解析 text/reasoning/tool
Content/tool preview conflation
Android frame batcher
Flutter reducer
局部 repaint
  • AgentLlmClient.kt 使用 AtomicReference 存最新 content 和每个 tool index 的最新 preview，保证高频 token 不堆积成无限 lambda 队列。
  • AgentStreamEventBatcher.kt 只对 text_snapshot, thinking_snapshot, 无 terminal delta 的 tool_progress 合并；tool_started/tool_completed/error/completed 保序。
  • streaming_text.dart 把已完成段落提交为稳定 Markdown widget，pending 段落随 token 更新，减少整段回答反复 parse/render。
  • AgentActivityCompactor 从全量重算改为 append-only 增量处理，工具卡片只保留同 card id 最新状态。

2. RunLog 收集和恢复

  • InternalRunLogStore.kt 每个 run 同时有 pretty snapshot 和 .events.ndjson append log。
  • running card 高频 upsert 只追加事件，snapshot 以最小间隔写入；完成、失败、非 running 状态强制写 snapshot。
  • 读取时先加载 snapshot，再应用 snapshot 之后的 event seq，解决进程退出或 snapshot 滞后导致的 timeline 丢失。
  • InternalRunLogStoreTest 覆盖 snapshot 后继续 append event、timeline 可看到最新 running card。

3. RunLog 到 Function 的编译

  • RunLogReusableFunctionCompiler.kt 从 card 的 tool_call/function/args/before/observation 提取步骤，输出 oob.reusable_function.v1。
  • RunLogReplayPolicy.kt 维护 canonical action 和 alias：例如 tap -> click, type_text -> input_text, scroll_* -> swipe, finish -> finished。
  • 失败 card 不参与“已有可回放动作”判断，避免失败 click 把 VLM-only fallback 跳过后生成空 Function。
  • android_privileged_action 中的本地 UI action 会从 arguments 展平成端侧执行器需要的 x/y/text/package_name 等字段。
  • 纯端侧 action 标记为 executor=omniflow 和 model_free=true；browser/search/memory/runlog 等 data-flow 工具保留 executor=agent。
  • provider-owned 图执行工具，例如 go_to_node, click_node, call_function，不降级成本地坐标脚本，统一标记为需要 OmniFlow/provider fallback。
  • Flutter 侧 RunLogReplayPolicy 与 Android asset replay_policy.json 有测试对齐，避免 UI 转换器和 native runner 分叉。

4. 端侧重放和坐标修正

  • OmniflowStepExecutor.kt 支持 click, long_press, scroll/swipe, type/input_text, open_app, press_key, wait, finished。
  • 如果有 source XML 和当前 XML，坐标动作通过 anchor projection 重新投影，避免录制时坐标在当前页面偏移。
  • ActionTargetGrounder.kt 对实时 VLM click/long_press 做二次修正：先用目标描述匹配节点文本，再结合原坐标附近性，命中后点击节点中心。
  • 低置信度、不相关目标、缺失 XML 时不强行改坐标，避免把 VLM 的原始意图破坏掉。

5. 外部 GUI Agent 接入

  • omniflow_agentkit 提供 prompt bundle、repo probe、MCP list/guard/run CLI。
  • scripts/omniflow_mock_mcp_server.py 提供无设备 mock MCP，验收外部 agent 是否真的能触发 oob_function_run。
  • omniflow_acceptance_all_guiagents.sh 覆盖 mobilegpt、mobile-use、mobile-mcp 三类 repo，分别验证 Python skill + MCP 和 direct MCP 模式。

6. 共享 Run Part 投影层

  • 参考 Anthropic content block、Vercel AI SDK message part、OpenAI Responses output item 的共同模型，新增 AgentStreamRunProjection：把 stream event 投影成稳定 run part，而不是在页面里临时补分支。
  • thinking_started 统一投影为空的 thinking part，占位先出现；后续 thinking_snapshot 只更新同一个 part，避免 output 先落屏、thinking 后插入导致顺序错乱。
  • 多轮 thinking 只清理升级到 *-thinking-2 等新轮次时遗留的空首轮占位；已有内容的 thinking round 不被删除。
  • error 是 run 的 terminal status，不是 assistant text part 的替代正文：已有用户可见 output 时保留正文；没有正文时才投影成错误 bubble。
  • 消息排序统一为 createAt + stream seq，解决同一毫秒内 thinking/text/tool/error 到达顺序不稳定的问题。
  • WebChat 已从局部 upsert 逻辑切到共享投影层；主聊天页新增直通 onAgentStreamEvent 回归测试，覆盖 thinking-start → thinking-snapshot → output → error 后仍归并为同一个 run。

验收标准

  • 通过：RunLog replay policy 在 Android asset、Android Kotlin、Flutter Dart 三处一致。
  • 通过：VLM-only 和 data-flow step 不被错误编译成本地直接重放。
  • 通过：canonical action alias 可以编译成 model-free OmniFlow step。
  • 通过：provider-owned graph/function step 会停留在 agent/provider fallback，不会误当成普通 tool replay。
  • 通过：失败的本地 replay card 不会吞掉 VLM fallback；android_privileged_action.arguments 会展平为可执行端侧参数。
  • 通过：ActionTargetGrounder 能把点击修正到匹配节点中心，低置信度不改动作。
  • 通过：AgentKit CLI 可通过 mock MCP 完成 list/guard/run。
  • 通过：共享投影层单测覆盖 error 不覆盖已有 output、无 output 才生成错误 bubble、同毫秒按 seq 排序、空首轮 thinking 占位清理。
  • 通过：直通 stream 在 error 后保留已输出正文，thinking 和 output 仍归并到同一个 Agent run。
  • 待真机：长流式回答 CPU/帧耗时、RunLog 大量 upsert 写入频率、真实设备 replay 触达率。

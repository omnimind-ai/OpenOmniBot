你描述需求 → AI 生成工具 → 工具留下来。

Project 解决的不是"AI 够不够聪明"，而是生成的东西能不能活下去。

Omniflow 持续沉淀你的习惯

⚡

对开发者意味着什么 架构革命

传统路径：需求 → 写代码 → 搭后端 → 部署 → 维护服务器。
小万路径：需求 → 描述规则 → 立即在设备上运行。后端这一层直接消失。

结构性壁垒

大厂为什么做不到这件事

能力 云端 AI（OpenAI / 百度 / 阿里） 手机厂商 AI（小艺 / 小布） 小万
端侧代码执行 ✕ 算力在服务器，进不了你手机 ✕ 系统 AI，不开放沙盒给用户 ✓ Alpine 沙盒本地跑
全局屏幕理解 ✕ 看不见用户手机屏幕 △ 有但不开放给第三方 ✓ VLM 实时读取任意界面
跨 App 操控 ✕ 没有设备权限 △ 仅限自家 App 生态 ✓ 无障碍服务读控任意 App
本地持久化数据 ✕ 只有会话记忆，服务端存储 ✕ 数据在厂商云端 ✓ 手机本地 /data，用户持有
用户自造工具 ✕ 生成代码，用户自己部署 ✕ 固定功能，不可定制 ✓ 描述即生成，即刻可用
根本原因 商业模式、生态封闭、或没有手机这个 surface——三方各有一个结构性做不到 唯一同时满足的产品

厂商 AI 有系统权限，但目标是让你用系统功能（订餐、导航），不是让你造自己的工具。这是消费者工具 vs 创作者工具的根本差异。

结构

一个 Project 由五层组成

🖥️

前端 UI 展示层

AI 生成的可视化界面（HTML / Flutter / Markdown）。用户看到的、交互的部分。
支持热更新——你圈一块 UI 说"改成这样"，它直接变。

🗄️

持久化数据 数据层

items + fields，跨会话留存。你的健身记录、账单、错题本……
不在云端，在你手机本地的 data/ 目录。

🔧

API / 工具 能力层

AI 为这个 Project 定义的可调用工具：calculate_tdee、log_meal、generate_workout……
前端调用 → 后端执行 → 结果写回数据层。

⚙️

后端执行 运行层

OOB 53 个原生工具 + Alpine 沙盒 + 小万系统能力（截图、无障碍、读写文件）。
真实在设备上运行，不是模拟。

🧠

Omniflow 记忆 学习层

把高频操作轨迹压缩为离线 Function。用得越多，调用大模型越少，速度越快成本越低。

生命周期

从一句话到持续演化的工具

01 · 创建

描述需求

用自然语言说你要什么。AI 理解意图，生成 UI + API + 初始数据结构。

02 · 激活

Project 上线

前端界面可以打开了。数据为空，等待你第一次使用。

03 · 使用

数据积累

每次操作写入 items。API 被调用，后端执行，结果可见。

04 · 编辑

随时热更新

悬浮窗圈选 UI + 说需求 → 界面实时改变。数据不丢。

05 · 进化

越用越便宜

Omniflow 把高频轨迹压缩，逐渐减少对大模型的依赖。

这五步构成一个正向飞轮——Project 不是一次性生成物，是一个活的工具。

实际案例 · 记账教练

Project 记住了你的规则，不只是数据

用户

「帮我做一个记账工具，自动分类我的消费，每月复盘。」

AI 生成

前端：消费分类图 + 月度趋势，API：import_bill / categorize_expense / generate_saving_plan。

日常使用

截图支付宝账单 → 小万读取 → 自动解析金额和商户 → 写入今日记录。

用户调整规则

「以后奶茶算娱乐，不算餐饮。」→ 分类规则更新，历史数据全部重新分类。

月底

「上个月哪里花多了？」→ 自动分析异常支出 + 生成下月建议。

规则是你的，数据是你的，UI 随你改——和现成记账 App 的本质区别：它按你的逻辑工作。

你做什么

  • 「这里改成折叠图表」
  • 「帮我加一个导出按钮」
  • 「这个颜色太亮了换深一点」
→
热更新
系统做什么

三件事同时发生

  • VLM 理解你圈的区域和意图
  • AI 重新生成对应的 UI 代码
  • 前端局部更新，其他数据不受影响

热更新不是重新生成整个 Project——是精确地改你指出的那一块。
你的数据、其他 UI、API 逻辑全部保持原样。

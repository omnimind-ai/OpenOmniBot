结论摘要

本次核心优化在实时 VLM 执行链路加入语义目标 grounding，把模型给出的点击/长按坐标与当前 Accessibility XML 中的可操作节点做语义和空间联合匹配，再将落点校正到目标中心。
可用性收益减少坐标偏移、缩放误差、按钮边缘点击、列表项父容器无文本等移动端常见失败；对无语义匹配的目标保持原坐标，不盲目吸附。
验证状态通过 Grounding、RunLog replay、Flutter converter 和 AgentKit MCP 单元验收。
剩余限制受限 完整真实 GUI 自动点击闭环仍需要连接设备并启用 OOB 无障碍服务。

当前 OOB GUI Agent 逻辑

当前代码里 GUI agent 主要有两条链路：实时 VLM 控制链路和 RunLog 可复用回放链路。

实时链路：VLMOperationService.executeTask 初始化上下文、可选启动目标应用、按步循环执行，支持 maxSteps 上限、暂停检查、上下文压缩和执行 trace。
每步执行：截屏和采集 XML，调用 VLMClient 流式请求模型，解析标准 tool call，映射坐标，做 XML 稳定性检查，再由 ActionExecutor 调用 DeviceOperator 点击、输入、滑动、返回、等待等动作。
工具协议：VLMToolDefinitions 定义 click/type/scroll/long_press/open_app/press_back/wait/info/finished 等动作，约束坐标必须是标量，降低模型输出格式漂移。
RunLog 链路：实时任务完成后写入内部 RunLog；OobFunctionToolHandler 优先走 deterministic Omniflow 回放，坐标重映射失败时再 fallback 到 vlm_task。

与 OmniFlow 的关系

结论：本次优化和 OmniFlow 是上下游互补关系，不是两套互斥方案。

层级 当前 OOB 代码位置 职责 本次优化关系
实时 GUI Agent assists/src/main/java/.../vlmserver/VLMOperationService.kt 看当前截图/XML，调用 VLM 选择下一步 click/type/scroll 等动作，并立即执行。 直接增强 在执行前用 ActionTargetGrounder 校准 click/long_press 坐标。
RunLog 记录 VLMOperationTask.appendInternalRunLog、InternalRunLogStore 把实时 agent 的每步 action、参数、summary、前后上下文写成可追踪 timeline。 间接增强 RunLog 记录到的是校准后的坐标和 [grounded:...] summary，后续转换更干净。
OmniFlow / Reusable Function app/src/main/java/.../runlog/、OobFunctionToolHandler.kt 把成功 RunLog 转成可复用 Function；回放时优先执行 deterministic local steps，失败再 fallback 到 Agent/VLM。 可复用 当前 OmniFlow 已有 OmniflowStepExecutor.remapStepArgs 基于 source/current XML 做 anchor projection；它解决“回放时页面变了”的问题。

所以边界应这样理解：

  • ActionTargetGrounder：用于“实时第一次执行”阶段，解决模型这一步给出的坐标不稳、按钮边缘、复合控件父容器无文本等问题。
  • OmniFlow remap：用于“历史 RunLog 重放”阶段，解决录制页面和当前页面布局不同，需要从 source XML 投影到 current XML 的问题。
  • 两者可以进一步统一为一个共享的 GuiTargetResolver：实时执行传入 target_description + currentXml + rawPoint，OmniFlow 回放传入 sourceXml + currentXml + recordedPoint + target_description。这样能避免 grounding 逻辑分叉。

当前仓库文档 app/src/main/assets/omniflow/runlog/README.md 也明确写了：OmniFlow 是从 RunLog 到 Function matching、execution 和 agent fallback 的 pipeline，不是单独的实时 GUI agent runtime 层。

对标公开 GUI Agent 设计

来源 关键设计 对 OOB 的启发
OpenAI Computer Use 截图驱动的 observe-act loop：模型看截图、返回点击/输入/滚动等结构化动作，执行后回传新截图继续决策；同时建议隔离环境、allow list 和高影响动作 human-in-the-loop。 OOB 已有截图+工具调用循环，应继续补齐动作后验证、风险动作自动确认、任务级可观测指标。
Anthropic Computer Use 提供截图、鼠标、键盘、沙箱环境和 agent loop；官方明确提示 beta 阶段存在延迟等限制，适合可信环境和后台任务。 OOB 移动端要把性能和权限状态纳入运行前诊断，避免用户以为 agent 卡死。
Microsoft OmniParser 把屏幕截图解析为结构化 UI 元素，强调可交互区域检测、元素语义理解，以及把意图准确绑定到屏幕区域。 OOB 有 Accessibility XML 天然结构化信号，应该在执行前用语义目标校准 VLM 坐标。本次已落地。
Google AndroidWorld Android GUI agent 通常每轮采集截图和 UI 元素，选择支持动作，执行并返回 done；动态任务和可靠 reward 用于评估泛化能力。 OOB 需要从“能跑”升级到“可量化可用”：记录任务成功率、步数、fallback 次数、误点率和权限失败率。
ByteDance UI-TARS 统一动作建模、GUI grounding、多步 System-2 推理、里程碑识别、反思式在线 trace 迭代。 OOB 已有 summary、compactor 和 RunLog，可进一步做 post-action verifier 和从成功/失败 trace 中自动改进 grounding/replay 策略。

主要问题与优先级

优先级 问题 影响 处理状态
P0 模型坐标在执行前主要依赖数值映射，target_description 没有参与落点校准。 按钮边缘、截图缩放、屏幕分辨率差异、动态列表位置变化时容易误点。 已优化 新增 ActionTargetGrounder。
P0 Android 页面常见“可点击父容器无文本、子 TextView 有文本”的结构。 只看可点击节点自身文本会漏匹配列表项、卡片按钮、复合控件。 已优化 grounding 合并子孙节点语义。
P1 已有执行前 XML 稳定性检查，但缺少动作后语义验证。 点击未生效、误跳页面、键盘遮挡等问题需要下一轮模型自救，成本高。 建议后续 增加 post-action verifier。
P1 高风险动作虽有 confirmation 工具，但没有统一风险策略拦截。 发送消息、支付、删除、提交表单等外部副作用动作不应只依赖模型自觉。 建议后续 增加 policy gate。
P1 缺少持续基准和真机任务集。 可用性提升难以量化，回归风险高。 建议后续 建立 AndroidWorld 风格 OOB smoke set。

本次代码优化

1. 新增语义目标 Grounding

新增 assists/src/main/java/cn/com/omnimind/assists/task/vlmserver/ActionTargetGrounder.kt。它解析当前 Accessibility XML，抽取可操作节点的 bounds、text、content-desc、hint、resource-id、class 和子孙文本，然后对 click/long_press 执行如下策略：

  • 仅处理 click/long_press，scroll 和非坐标动作不改。
  • 语义相似度和空间邻近度联合打分，目标必须有足够语义匹配，不做纯坐标吸附。
  • 过滤过小区域和大面积容器，避免吸到整屏根节点。
  • 命中后把坐标校正到目标节点中心，并把原因写入 summary：[grounded:inside_target_centered] 或 [grounded:semantic_target_centered]。
  • 支持可点击父容器读取子孙文本，覆盖真实 Android 复合控件结构。

2. 接入实时执行链路

在 VLMOperationService.executeSingleStep 中，坐标映射后、XML 稳定性检查前执行 groundActionTarget(processedStep, beforeXml)。这保证执行稳定性检查和真实点击使用的是已校准坐标。

3. 编译阻塞修复

app/src/main/java/cn/com/omnimind/bot/runlog/OmniflowStepExecutor.kt 中 RunLog 滑动距离字段统一为 Float，避免 debug APK 编译失败。

4. 单元测试

新增 assists/src/test/java/cn/com/omnimind/assists/task/vlmserver/ActionTargetGrounderTest.kt，覆盖：

  • “发送”按钮坐标校正到节点中心。
  • “删除”不应误吸附到“发送”。
  • “张三”长按目标校正。
  • “确认支付”可点击父容器依赖子 TextView 文本仍可命中。

后续优化路线

阶段 目标 建议实现 可用性指标
短期 动作后验证 每次 click/type/scroll 后采集 XML 和截图，判断目标状态是否变化；未变化时重试 grounding 或请求 VLM 重新定位。 误点后自恢复率、平均额外步数。
短期 权限前置诊断 任务开始前检查无障碍、悬浮窗、截图权限、目标应用安装状态，失败时给出可操作提示。 权限类失败率、首步失败率。
中期 风险动作 policy gate 基于 action + target_description + 当前包名识别“发送、支付、删除、提交、发布、授权”等动作，强制走用户确认。 高风险动作未确认执行次数应为 0。
中期 RunLog 质量闭环 把 grounding 命中率、fallback 原因、post-action verifier 结果写入 RunLog，供回放和调试。 可回放成功率、fallback 次数。
长期 AndroidWorld 风格评测 维护 20-50 个真机/模拟器 smoke tasks，覆盖打开应用、搜索、填写表单、列表滚动、跨应用跳转、用户确认。 任务成功率、P50/P95 步数、P50/P95 耗时。

一句话给别人讲

main 还是以通用 AI 助手、手机自动化、聊天和基础工具为主；wzw-dev 增加了 Workbench Project、OmniFlow 可复用指令、RunLog 回放、外部 MCP 调用和更清晰的 Agent 执行展示，目标是让一次成功的 Agent 工作可以被看见、保存、复用和分享。

产品重心

从“完成一次任务”转向“沉淀成用户可持续使用的小工具和自动化能力”。

技术边界

把 Project、Function、RunLog、MCP、GUI replay 拆成稳定运行时，而不是只靠对话过程临时拼接。

外部价值

其他 GUI Agent 可以通过 OmniFlow kit/MCP 复用 OOB 记录下来的手机执行能力。

从一次 Agent 任务到可复用资产

wzw-dev 新增的核心闭环可以理解成下面这条产品链路。它解释了为什么这不是单点功能，而是一个新平台层。

01 用户提出需求

从 Home/聊天进入 Project 创建、手机操作、网页研究或工具调用。

02 Agent 执行并记录

过程通过统一 stream 事件、tool card、RunLog 和 VLM trace 留痕。

03 生成 Project 或 Function

结果可以成为可视化 Project，也可以把成功路径转成可复用指令。

04 安全回放和热更新

OmniFlow guard 判断 allow、needs_agent、needs_confirmation 或 block。

05 外部 Agent 调用

MCP 暴露 recall、call_function、ingest_run_log 和 Project Toolbox。

核心功能展开

如果只向业务或产品同学解释，建议聚焦这五块。

1. Workbench Project：从聊天输出变成持久工具

wzw-dev 把 Project 作为 OOB 内置核心能力：Project 有自己的数据、展示、业务 API、日志、导出包和活跃状态。

  • 支持 workbench_project_create/list/get/update/open/activate/export/delete 等控制面能力。
  • HTML WebView 成为默认富展示路径，同时保留 Markdown、Flutter eval 和通用 Project Display。
  • 支持 hot update：用户在页面上说“改这里”，Agent 带着当前前端上下文迭代同一个 Project。
  • 支持 OSS/Android 资产导入，为“把现有项目蒸馏成 OOB Project”铺路。

2. OmniFlow：把成功执行沉淀成可复用 Function

这是 wzw-dev 最重要的自动化能力：不是每次都让模型重新规划，而是把成功 RunLog 转成结构化 Function。

  • Function schema 固定为 oob.reusable_function.v1，包含参数、执行步骤、来源和 guard 元数据。
  • 低风险 UI 动作走本地 replay；浏览器、搜索、记忆、VLM-only 等步骤标记为 Agent fallback。
  • 危险动作有 guard：本地 UI replay 允许，shell/设置/安装/权限类需要确认，重启/格式化等阻断。
  • 指令库让用户直接查看、执行和删除已保存 Function。

3. RunLog 与执行历史：从日志变成产品入口

main 里的执行历史更像记录；wzw-dev 让它成为 replay 和审计入口。

  • 新增 InternalRunLogStore，统一保存 Agent/VLM/工具执行过程。
  • 新增 RunLog 列表、时间线、步骤详情、token/耗时等展示。
  • RunLog 可转换成 Function，并可通过 UI、Agent tool 或 MCP 再次执行。
  • 文本型 Agent run 也会保留 RunLog，避免只包含工具步骤才可追踪。

4. Agent Stream UI：长任务更可读

wzw-dev 用统一事件流重做 Agent 过程展示，让聊天页、悬浮窗、RunLog timeline 共享同一套投影。

  • 后端输出 oob.agent_event.v1 事件 envelope，包含 trace/run/span/channel/status 等字段。
  • Native 侧按帧批量发送 stream 事件，降低高频 token 和工具进度带来的 UI 压力。
  • 前端用 reducer 和 compactor 把浏览器、终端、Workbench、MCP 等调用聚合成活动卡。
  • 新增运行中 Agent 悬浮监控，能看到活动状态并处理停止/展开等操作。

5. 外部接入：OOB 能力可以被别的 Agent 调用

wzw-dev 不只服务 App 内部。OmniFlow kit 明确给外部 GUI Agent 一套接入契约和验收方式。

  • MCP 暴露 canonical 工具：omniflow.recall、omniflow.call_function、omniflow.ingest_run_log、omniflow.explore_replay。
  • 外部可以 list/get/register/guard/run Function，也能 list/get/convert RunLog。
  • 文档和脚本覆盖 MobileGPT、mobile-use、mobile-mcp 这类外部项目验收。
  • Project Toolbox 让 active Project 的业务动作也能通过 MCP 动态暴露。

6. GUI replay 基础：VLM 操作更容易复现

wzw-dev 在 VLM/Accessibility 层补了可记录、可 grounding、可探索的基础设施。

  • 记录每次 VLM task 的原始操作 trace，用于后续 RunLog/Function 物化。
  • UTG explorer 从 Accessibility XML 中抽取可 replay 路径。
  • coordinate hook 和 source_context 帮助点击坐标在新页面上下文中重新定位。
  • 截图 fallback、Assists rebind、VLM compile metadata 等稳定性也有加强。

main 和 wzw-dev 的定位差异

main：通用 AI 助手主线

当前 main 更像稳定的 Omnibot 基线：聊天、Agent、手机 VLM 操作、基础工具、设置、历史和现有 Flutter UI。它能完成一次任务，但“任务完成之后如何沉淀成资产”的产品链路还不完整。

wzw-dev：Project + OmniFlow 平台层

wzw-dev 叠加了一层面向持续使用的产品基础设施：Project 管理可视化结果，RunLog 记录过程，Function 复用成功路径，OmniFlow/MCP 让内部和外部 Agent 都能安全调用。
